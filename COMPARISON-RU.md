# booking_widget (Vue 3) → booking_widget_elm (Elm)

🇬🇧 [Read in English](./COMPARISON.md)

Параллельная переделка того же виджета бронирования номера, отвечающая на один
вопрос: **сколько дыр в корректности оригинала закрывает чистая TEA-архитектура на
Elm — структурно, а не «не забыл проверить»?**

Тот же домен, те же детерминированные данные о занятости, тот же поток (даты →
гости → номер → проверка → подтверждение), та же i18n ru/en, переключение темы,
календарь и граница данных mock/http. Разница — целиком в том, *как смоделировано
состояние и как выражены переходы*.

- Vue-версия: [репозиторий](https://github.com/fellz/booking_widget) · [демо](https://fellz.github.io/booking_widget/)
- Effect/foldkit-версия: [репозиторий](https://github.com/fellz/booking_widget_effect) · [демо](https://fellz.github.io/booking_widget_effect/)
- Elm-версия: этот проект (`src/`) · [демо](https://fellz.github.io/booking-widget-elm/)

Запуск: `npm run dev`. Проверка: `npm test` (77 тестов), `npm run typecheck`, `npm run build`.

---

## Архитектура: полноценный TEA

Elm *и есть* The Elm Architecture — тот самый цикл, который соседняя
Effect/foldkit-версия переизобретает на TypeScript; здесь это единственный способ
построить приложение:

- **Model** — всё состояние приложения в одной неизменяемой записи
  (`src/Model.elm`). Никакого локального состояния компонентов.
- **Msg** — каждый ввод (клики, результаты команд) — вариант одного
  пользовательского типа (`Model.Msg`). Ничего не происходит, что не было бы `Msg`.
- **update** — единственная *чистая* функция `Msg -> Model -> (Model, Cmd Msg)`
  (`src/Update.elm`), разбираемая `case`, исчерпываемость которого проверяет
  компилятор. Единственное место переходов состояния, и оно не делает I/O.
- **view** — чистая функция `Model -> Html Msg` (`src/View.elm`). UI — это
  проекция Model; собственного состояния держать не может.
- **Cmd** — побочные эффекты — это *значения*, возвращаемые из `update`,
  исполняемые рантаймом Elm, который подаёт их результаты обратно как `Msg`
  (`src/Api/`). View никогда не вызывает API; он диспатчит `Msg`, `update`
  возвращает `Cmd`.

`Browser.element` (`src/Main.elm`) замыкает цикл: `Flags` для входных данных старта
(`today`, тема, api url) и два `port` для единственных записей в DOM за пределами
Elm-корня (тема на `<html>`, `<html lang>`).

## Аудит

Аудит корректности Vue-виджета нашёл **13 дыр** (тот же набор, по которому
оценивается Effect-версия). «Закрыто структурно» = плохое состояние невыразимо в
типе/модели, либо переход, который его породил бы, — это ошибка компиляции или
требование исчерпываемости, а не соглашение, которое будущая правка может тихо
сломать.

| # | Дыра (Vue) | Важн. | Закрыто | Как Elm её закрывает (или нет) |
|---|------------|-------|---------|--------------------------------|
| 4.1 | `DateRange = { checkIn, checkOut }` мог хранить `checkOut` при `checkIn = null` | LOW-MED | ✅ структурно | `Selection = NoSelection \| CheckIn BookableDate \| Range BookableDate BookableDate`. Полузаполненное состояние не представимо. `Range` строится только при `checkIn < date`, поэтому «заезд раньше выезда» держится по построению. (`Domain.Types`, `Update.selectDate`) |
| 2.1 | `status: editing\|review\|confirmed` — флаг, *оторванный* от валидности, согласованный лишь условным рендером | MED | ✅ структурно | В `Phase` нет голого флага review/confirmed: каждый пост-редактирующий вариант (`Reviewing`/`Submitting`/`SubmitFailed`/`Confirmed`) **несёт** `ValidBooking`, построенный только умным конструктором `validBooking`. Нельзя получить подтверждённую бронь без валидированной полезной нагрузки; экраны проверки и подтверждения берут её, поэтому «подтверждено, но показывать нечего» невыразимо. (`Model.Phase`, `View`) |
| 3.1 | Бронь размазана по 4 ref + 1 reactive; межполевые правила («сбросил номер ⇒ очисти календарь») держались лишь потому, что каждый мутатор не забывал их вызвать | MED | ✅ структурно | Одна неизменяемая `Model`, один чистый `update`. Каждый переход возвращает целостное состояние целиком; второго писателя, который мог бы забыть правило, нет. |
| 7.1 | Неисчерпывающие тернарники `locale === 'ru' ? … : …` в 3 местах — третья локаль молча форматировалась бы как `en-GB` | MED | ✅ структурно | Локаль разбирается `case` по типу `Locale` в каждом месте форматирования (`messages`, `errorMessage`, `currencyCode`, названия месяцев…). Новая локаль — ошибка компиляции в каждом. (`I18n.elm`) |
| 1.1 | У `loadRooms` не было инвалидации запроса → медленный/повторный ретрай мог затереть свежие номера или сбросить статус в error при наличии данных | HIGH | ✅ структурно | Обе загрузки несут монотонный request id; `update` отбрасывает любой `RoomsLoaded`/`CalendarLoaded`, чей id ≠ текущему в модели. Во Vue это было только для календаря; здесь — единообразно. (`Update.elm`) |
| 5.3 | Ошибка загрузки календаря = задизейбленная кнопка, без подсказки и ретрая (ветка ошибки вообще не рендерилась) | MED | ✅ структурно | `CalendarState` включает `CalError`; `case` в легенде исчерпывающий, поэтому состояние ошибки *нельзя* молча пропустить (во Vue — пропускалось). Оно вынуждает видимый, восстановимый баннер с действием `RetryCalendar`. (`Model.CalendarState`, `View.calendarLegend`) |
| 1.2 | `today` календаря (время фетча) мог разойтись с `today` пикера (время монтирования) через полночь | LOW | ✅ структурно | Один `today` в модели, взятый один раз из стартового `Flag`. Одно значение, используется везде. |
| 4.2 | `calendarStatus === 'ready'` был представим без выбранного номера, и «ready» не гарантировал наличие данных | LOW | ✅ структурно | Доступность *вложена в выбор*: `RoomSelection = NoRoom \| RoomChosen RoomId CalendarState`. Без номера поля календаря нет вообще, а `CalReady from to (Set String)` несёт данные — и «ready без номера», и «ready без данных» невыразимы. (`Model.RoomSelection`/`CalendarState`) |
| 7.2 | `goToStep` зашит `index === 0 ⇒ edit()`; перестановка шагов тихо ломает навигацию | LOW | ✅ структурно | `Ui.steps` больше не принимает `onSelect : Int -> msg`; каждый шаг несёт свой `onBack : Maybe msg`, определяемый исчерпывающим `backMsg : Phase -> Step -> Maybe Msg` по именованному `Step = ChooseStep \| ReviewStep \| DoneStep`. Навигация — по *идентичности* шага, а не по позиции: проверки `index == 0` нет, и сообщение `GoToStep` удалено. (`Ui.steps`, `View.backMsg`) |
| 6.1 | Нет отмены запросов; устаревшие фетчи завершались и просто игнорировались | LOW | ⚠️ частично | Теперь отменяются по-настоящему: HTTP-адаптер шлёт `getRoomCalendar` с `tracker = Just "room-calendar"` и `cancelCalendar = Http.cancel …`, а `update` прерывает запрос в полёте при supersede/снятии выбора/сбросе. Гард `requestId` остаётся как defense-in-depth. **Не** структурно: устаревший ответ нельзя *запретить* типом — отмена это рантайм-эффект, поэтому «улучшено», а не «закрыто по построению» (моку нечего прерывать → `Cmd.none`). (`Api/Http.elm`, `Update.startCalendarLoad`) |
| 2.2 | `confirm()` ставил флаг и не делал **никакого эффекта** — главная операция не могла ни упасть, ни сообщить об ошибке | MED | ✅ структурно | Confirm диспатчит настоящий эффект `submitBooking`; `Phase` получает `Submitting`, `SubmitFailed ValidBooking SubmitError` и `Confirmed ValidBooking String` (референс с бэкенда). Нельзя попасть в `Confirmed` без референса из эффекта. Сбои идут по отдельному `SubmitError = SubmitNetwork \| SubmitServer Int \| RoomTaken` (409 → `RoomTaken`), и i18n рендерит **разное сообщение на каждую причину** (исчерпывающий `submitErrorMessage`). | 5.1 | Заезд за горизонт фетча в 180 дней считался **доступным и бронируемым** — у множества занятости просто не было данных на те дни | HIGH | ✅ структурно | `CalReady` несёт *окно* `from`/`to`, которое реально покрывает. `availabilityErrors` помечает любой заезд за этот несомый `to` как `AvailabilityUnknown` (не бронируется), а не молча свободный, и навигация по месяцам зажата до месяца горизонта (кнопка «вперёд» дизейблится). «Бронируемо без данных» не возникает ни через UI, ни проходя валидацию. (`Model.CalendarState`/`availabilityErrors`/`maxMonthCursor`) |
| 5.2 | Правило прошедших дат жило **только** в UI календаря (`isPast`); домен принимал прошлые даты из любого другого пути | MED | ✅ структурно | Правило переехало в *тип*. Непрозрачный `Domain.BookableDate` строится только `fromDate today candidate`, который отвергает прошлый день; `Selection` и `ValidBooking` несут `BookableDate`, поэтому прошлый заезд **невыразим** — нет значения, чтобы положить в модель, и нет ошибки `PastDates`, чтобы её забыть (она убрана из `BookingError`). `validateBooking` больше не принимает `today`. Цена: церемония wrap/unwrap на границах (`toDate` для рендера и отправки). (`Domain.BookableDate`, `Domain.Types`, `Update.selectDate`) |

**Итог: 12 из 13 закрыты структурно, 1 закрыта на практике (отмена — рантайм-
эффект, а не тип), 0 осталось открытыми.**

Честный вывод: дыры представления и исчерпываемости — невозможные состояния,
оторванные флаги, неисчерпывающие матчи, размазанное состояние — закрываются
чисто сумм-типами и тотальным `update`. Главное здесь — 2.1/2.2: `Phase` проносит
smart-constructed `ValidBooking` через проверку, отправку, сбой и подтверждение,
поэтому ни один из этих экранов не отрисуется без данных, которые его оправдывают;
у write-стороны свой типизированный `SubmitError` с разным сообщением на причину.
Последняя тройка пошла туда же: 5.2 перенесла правило прошедших дат в непрозрачный
`BookableDate` (прошлый заезд теперь невыразим, а не просто перепроверяется), а
7.2 заменила позиционную back-навигацию `index == 0` на пошаговый `onBack`,
определяемый исчерпывающим матчем `Phase × Step`. **Единственная** дыра, что
осталась «на практике», — 6.1: устаревшие запросы календаря теперь реально
отменяются (`Http` `tracker` + `Http.cancel`), но устаревший ответ нельзя
*запретить* типом — отмена это рантайм-эффект, поэтому реальным механизмом
корректности остаётся гард `requestId`. По аудиту из 13 дыр Elm теперь чуть
впереди (Elm 12/1/0, Effect 9/4/0). Где Effect-версия действительно сильнее — это
**вне** этого списка: её Schema *и есть* доменный тип на границе IO (декодер и тип
не разъедутся — см. ниже), а DI выражен типизированным графом `Layer`, а не одним
рекордом-адаптером. Это реально и в ванильном Elm недостижимо.

## Две победы вне списка из 13 дыр

В README оригинального Vue были также перечислены пробелы типизации, не связанные
с TEA-состоянием. Две крупнейшие Elm закрывает по построению:

**1. Граница ввода-вывода — декодируется, а не «as».** Vue-`httpBookingApi` делал
`(await res.json()) as RoomDto[]` и `dto.id as RoomId` — `as` выключает проверку,
поэтому неизвестный id, пропущенное поле или иная валюта компилировались, а падали
позже (`roomImages[room.id]` → битая картинка, без ошибки типов). Elm-HTTP-адаптер
прогоняет каждый байт через `Json.Decode.Decoder`, а `roomIdDecoder` *проваливает
декодирование* на неизвестной категории вместо порождения битого `RoomId`
(`src/Api/Http.elm`). Плохой вход не доходит до домена. Сбои идут по
типизированному каналу `ApiError` (`NetworkError | BadStatus Int | DecodeError
String`), а не схлопываются в один булев `'error'`.

> Оговорка, общая для любого подхода с рукописными декодерами: декодер и доменный
> тип — *отдельные* объявления и могут разъехаться (Schema-as-type в
> Effect-версии этого избегает). Здесь они держатся в одном небольшом файле и
> проверяются тайпчекером в точке вызова, но это соглашение, а не гарантия.

**2. Часовые пояса — устранены, а не нормализованы.** Vue-код нормализовал `Date` к
локальной полуночи повсюду (`startOfDay` оборонительно зван в `addDays`,
`daysBetween`, …), чтобы обойти баги времени суток, а README отмечал рассинхрон
TZ-ключей как латентный инвариант. `Date` из `justinmimbs/date` в Elm — это чистый
календарный день (целое rata-die): без времени, без зоны. Весь этот класс багов и
оборонительная нормализация исчезают. Единственная граница зоны — превращение
«сейчас» в `Date` один раз на старте (`src/main.ts` → флаг); в сравнения она
больше не входит.

## Чего это стоило

Закрытия оплачены **предварительным моделированием**, во многом как в
Effect-версии, но более лёгким словарём:

- Больше именованных состояний: `Selection`, `RoomsState`, `CalendarState`,
  `Phase` плюс smart-constructed `ValidBooking` — состояния, которые Vue-стор вёл
  неявно по нескольким ref.
- Никакого `Intl`: валюта, названия месяцев и русские склонения форматируются
  вручную (`src/I18n.elm`). Плюс — они чистые и покрыты тестами; минус —
  форматирование приблизительное (например, фиксированный неразрывный пробел как
  разделитель тысяч), не ICU-точное.
- View — многословный `Html`-DSL с самодельным UI-kit вместо Vue-`<template>` +
  scoped CSS (сам CSS переиспользован дословно).

Выгода: баги представления и исчерпываемости, которые аудит *нашёл* во Vue-версии,
здесь по большей части — то, что нельзя выразить или нельзя забыть; весь поток
покрыт быстрым детерминированным `elm-test` (юнит, property-based по датам и
«истории» по чистому `update`; без DOM, без моков таймеров).

## Тесты (без DOM, без моков)

Поскольку TEA разделяет чистый `update`, `view` и эффекты-значения, логика
тестируется изолированно — `tests/UpdateTest.elm` прогоняет настоящий `update`
последовательностями сообщений и проверяет получившуюся модель (машина состояний
дат, защита по request-id для обеих загрузок, межполевое правило номер/гости,
гейты фаз, ретрай календаря). `tests/DomainDateTest.elm` — property-based (fuzz) по
календарному ядру; `tests/BookingTest.elm` и `tests/I18nTest.elm` фиксируют чистые
правила и ручное форматирование. 77 тестов, ~120 мс.

## Где закрыты дыры (карта файлов)

- Невозможные состояния убраны → `src/Model.elm` (`Selection`, `RoomsState`, `CalendarState`, `Phase`, `ValidBooking`) и `src/Domain/Types.elm`
- Прошлые даты невыразимы (непрозрачный умный конструктор) → `src/Domain/BookableDate.elm`
- Единый источник валидности + умный конструктор → `Model.validBooking` / `Model.isValid`
- Навигация шагов по идентичности (пошаговый `onBack`) → `src/Ui.elm` (`steps`) / `View.backMsg`
- Защита по request-id + реальная отмена календаря, тотальные переходы → `src/Update.elm`
- Эффекты-значения, декодированная граница, типизированные ошибки → `src/Api/` (`Types.elm`, `Mock.elm`, `Http.elm`)
- Календарное ядро (убивает класс TZ) → `src/Domain/Date.elm`
- Исчерпывающая обработка локали → `src/I18n.elm`
- Поведенческое доказательство → `tests/`
